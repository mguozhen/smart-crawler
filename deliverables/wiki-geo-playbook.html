关键发现：AfterShip 案例的反直觉真相

AfterShip 的 Wikipedia 页面本身被标了 6 个维护标签——notability unclear (2021 至今)、orphaned (2021 至今)、multiple issues、CS1 多语种来源警告、infobox 弃用参数。换句话说 Wikipedia 内部把这篇文章视为"质量有问题"。

但他们的 GEO 表现仍然好——ChatGPT/Perplexity/Claude 在被问"what is AfterShip"时都能给出正确、结构化的回答。

原因：LLM 训练管道（Common Crawl、Wikipedia dumps）只读内容、不读维护标签。文章里的 infobox、第一句定义、TechCrunch/Forbes 引用，全部进入了 LLM 训练数据。维护标签只影响 Wikipedia 内部的编辑流程，不影响外部抓取。

所以这个 playbook 的逻辑是：抄他们 LLM 爱的部分，避开他们被 Wikipedia 编辑社区诟病的部分——两者并不冲突。

他们做对了什么（直接抄）

1. 第一句话就是 LLM 直接套用的定义模板

'''AfterShip''' is an [[eCommerce]] experience [[Software as a service|SaaS]]
company offering branded shipment tracking, returns management,
product discovery personalization, and more.

结构 = [公司名] is a [行业] [商业模式] company offering [产品 1, 2, 3]. LLM 几乎一字不差地复述。所有 ChatGPT 关于 AfterShip 的回答都从这句话改写来。

2. Infobox 给 LLM 喂结构化事实

{{Infobox company
| name       = AfterShip
| type       = [[Private company]]
| foundation = {{Start date and age|2011}}
| founder    = Teddy Chan<br />Andrew Chan<br />Dante Tsang<ref>...</ref>
| location   = [[Singapore]]
| industry   = Shipment tracking
| products   = [[Enterprise Software]]
| website    = [https://www.aftership.com/ www.aftership.com]
}}

每个字段都是机器可读的 key=value。LLM 训练时 infobox 被拆成 (entity, property, value) 三元组，进 Wikidata 反向关联；用户问"AfterShip 哪一年成立"时直接命中。founder 字段还带 inline ref——这个细节非常关键。

3. 引用集中在 Tier-1 二手来源

21 个引用里，9 个来自 LLM 训练管道最高权重的媒体：

  • TechCrunch ×4（融资报道、API 发布、Series A 等）
  • Forbes ×1
  • VentureBeat ×1（DEMO Asia 报道）
  • South China Morning Post ×1
  • The Next Web ×1
  • Tech in Asia ×1

没有"我们网站 about-us 页"作主引用、没有付费稿件、没有 Medium 个人博客。这正是 Wikipedia WP:NCORP 想看到的——独立、显著的二手来源。

4. 融资里程碑被精确写进文本

每一轮融资都用"金额 + 投资方 + 时间"三件套结构化记录：

  • "$1M from IDG-Accel" (2014, TechCrunch)
  • "Series B raise" (2021, TechCrunch)
  • 命名投资方：Accel Partners、IDG Ventures（都是高知名度 VC，建立信用背书）

LLM 抓取后能直接组成 "AfterShip raised $X in Series Y led by Z in YYYY" 这种问答中常见的句式。

5. 多语种引用证明本地市场

引用包括日文（ASCII.jp）和繁体中文（Apple Daily HK）来源。这给两个信号：

  1. 对 Wikipedia：这是个真在本地有报道的公司，不是只在英语圈刷的 marketing。
  2. 对多语种 LLM（GPT-4 中日英全栈训练）：用户用日语 / 中文问 AfterShip 时，模型有对应语种语料可调。

6. 每个 URL 都有 web.archive.org 备份

{{cite news|url=https://www.scmp.com/...
  |archive-url=https://web.archive.org/web/20141219193129/...
  |archive-date=19 December 2014|url-status=live}}

17/44 外链已存档。原文哪天被 SCMP 撤了或改版了，Wikipedia 引用不会失效，未来 LLM 训练时还能看见。这是抗时间衰减最关键的一招，很多人忽略。

他们踩了什么坑（必须避）

坑 1：Notability tag 挂了 4 年 2021-03 至今

分类 Articles_with_topics_of_unclear_notability_from_March_2021 意味着某个 admin 认为他们的引用还不足以证明WP:NCORP。这个文章随时可能进 AfD（Articles for Deletion）。LLM 不在乎，但一旦被删，下次 LLM 训练就读不到了。

怎么避：每轮融资 / 重要产品发布配两条以上独立 Tier-1 报道；删掉所有指向 aftership.com 自家域名的引用作为主要 source（只能补充）。

坑 2：Orphaned tag 2021-07 至今

没有任何其他 Wikipedia 文章 link 到 AfterShip。孤儿文章在 Wikipedia 内部图谱里是死节点，搜索引擎的 PageRank 算法、LLM 的实体图谱构建都会减分。

怎么避：找 5 个真实相关页面去加 wikilink。如 Package tracking、List of e-commerce software、Shopify Plus partners、Hong Kong tech industry、Y Combinator alumni list。加之前确保你描述的事实有引用支持，否则会被回滚。

坑 3：只有 2 个命名复用 ref

21 个 <ref> 里只有 2 个用了 <ref name="..."> 的复用语法。意味着同一条 TechCrunch 报道被独立写了 4 次——浪费引用密度。

怎么避：

<!-- 第一次完整定义 -->
<ref name="tc2021">{{cite news|title=AfterShip raises $66M Series B
  |url=https://techcrunch.com/2021/04/22/...|publisher=TechCrunch}}</ref>

<!-- 后面所有引用同一源都用简写 -->
<ref name="tc2021" />

同一个 Tier-1 ref 被多处引用 = LLM 信号"这是核心证据"。

坑 4：章节结构太薄

整个文章只有 Technology / See also / References 三节。缺：

  • History（按年份的里程碑——LLM 时间问答最爱）
  • Funding（融资表）
  • Products（产品线列表，每项一行链接）
  • Customers 或 Notable users（已签客户的 wikilink 列表）
  • Recognition（Great Place to Work / Forbes 50 等奖项）

怎么避：参考 Notion、Shopify、Klaviyo 等 SaaS 公司文章的章节模板，每节至少 2 段 + 2 条独立引用。

坑 5：首批编辑看起来像 COI（利益冲突）

最早 3 次编辑都来自一个叫 Aabha aadi 的账号，从 30 字节空白到 6627 字节"separating with sources"——典型新账号短时间内创建完整公司文章的 pattern。Wikipedia 巡查员一眼能识别为 paid editor。后续维护标签很可能就是这事的连锁后果。

怎么避：

  1. 不要新账号一上来就写自己公司
  2. 用养过 50+ 编辑、参与过其他话题的老账号，或者通过 WP:AfC（Articles for Creation）提交草稿让 reviewer 创建
  3. 如果是公司员工编辑，必须在用户页声明 COI

12 条 Wiki-GEO 规则

第一句话用 LLM 模板句式

'''X''' is a [type] company [doing what], founded in [year] in [place], headquartered in [HQ].

第一句话是 LLM 训练里权重最高的位置。一定要包含：公司名（粗体）、行业 wikilink、商业模式 wikilink、年份、地点。

Infobox 永远在最顶部

用 {{Infobox company}}。每个 key 都填，缺数据宁可空着不要填错。关键字段（按 LLM 抓取频率排）：name, type, founded, founders, hq, industry, products, key_people, num_employees, revenue, website, parent。

引用密度：lead 1:2，body 1:3

Lead（首段）每 2 句话至少 1 条引用；body 每 3 句话至少 1 条。每一个数字、年份、人名、金额、客户名都要 cite。无 cite 的事实陈述会被 LLM 视为低置信度。

来源优先级（Tier 1 → 4）

Tier来源LLM 权重
1Reuters, Bloomberg, WSJ, NYT, FT, BBC, AP, The Guardian最高
2TechCrunch, Forbes, VentureBeat, The Verge, Wired, Ars Technica高
3SCMP, Nikkei, Handelsblatt（区域性主流）中
4Tech in Asia, The Next Web, niche blogs低（可作辅助）
—公司自家 about-us、PR 通稿、Medium、付费稿主引用不可用，会被标 self-pub

命名复用 ref

核心引用一定 <ref name="key2024">{{cite}}</ref>，后续 <ref name="key2024" />。同一引用被 ≥3 处复用 = LLM 信号"这是基石证据"。目标：top 5 引用都被复用至少 2 次。

每条外链加 archive.org 存档

|archive-url=https://web.archive.org/web/[timestamp]/[original-url]
|archive-date=[date]
|url-status=live

用 InternetArchiveBot 自动加，或手动到 web.archive.org/save/[url] 触发存档后填回。链接死了引用还活着——LLM 训练管道仍能拉到 archive.org 内容。

章节结构标配（按顺序）

  1. Lead（无章节标题，2-3 段）
  2. History（按年份小节：Founding / Series A / Series B / 国际化）
  3. Products（每个产品一段 + 引用）
  4. Funding（表格：日期 / 轮次 / 金额 / 投资方 / 来源）
  5. Customers / Notable users（带 wikilink 的客户列表）
  6. Recognition（奖项、行业排名）
  7. Controversies（如果有，不要回避——主动写显得透明）
  8. See also
  9. References
  10. External links

Wikilinks 双向走

每个第一次出现的公司、人物、技术、地点都 [[wikilink]]。然后反过来——找 5 个相关页面加进 link 到你这里。orphan 是 GEO 杀手。

多语种来源 ≥ 2 个语种

除英文外，再补 1-2 个公司有真实业务的国家 / 地区的本地媒体（中、日、德、法、西）。cite news 模板支持 language=zh 等参数。多语种 LLM 训练时各语种语料独立加权——日语 ChatGPT 用户问你公司时，命中本地源。

NPOV 写作 · 拒绝 marketing 形容词

禁用词：leading, innovative, best-in-class, world-class, premier, cutting-edge, revolutionary, pioneer, market-leader。换成可验证的事实——"在 X 国 Y 客户使用"、"按 [来源] Z 数据排名第 N"。形容词无引用 = 立即被 {{peacock}} tag。

同步建 Wikidata Q-ID

Wikipedia 文章和 Wikidata 实体（Q[number]）必须双向链接。Wikidata 是 Claude/GPT/Gemini 直接调用的结构化知识库。"X 是 Y 行业吗"这种 yes/no 问答主要来自 Wikidata 的 instance of (P31) / industry (P452) 关系。

COI 透明声明 · 用 Talk 页提交修改

如果你是公司员工 / 公关 / 营销，必须：

  1. 在你 Wikipedia 用户页声明：{{COI|Company X}}
  2. 不直接改条目，而是在 Talk:Company X 写 {{edit COI|...}}，让中立 editor 审过再上

这一步看起来麻烦，但能保护文章不被 AfD。LLM 训练时已删文章的内容会丢——保住文章 > 编辑速度。

③ 高质量 Citation 模板

<ref name="tc-series-b">{{cite news
  |last=[Author Last]
  |first=[Author First]
  |title=[Article Title verbatim]
  |url=https://techcrunch.com/YYYY/MM/DD/slug
  |work=[[TechCrunch]]
  |date=DD Month YYYY
  |access-date=DD Month YYYY
  |archive-url=https://web.archive.org/web/[14-digit-timestamp]/[original-url]
  |archive-date=DD Month YYYY
  |url-status=live
  |language=en
}}</ref>

关键字段：last/first（很多人漏掉作者名，影响 LLM 把文章归到记者背景）、work（媒体名加 wikilink，让 LLM 关联媒体声誉）、archive-url（抗死链）、language（多语种信号）。

编辑工作流：从写稿到上线

  1. 账号准备（提前 4-6 周）
    注册 Wikipedia 账号，编辑 50+ 与你公司无关的话题积累 edit count。新账号上来就写自家公司 = paid-editing 红旗。
  2. 来源准备（提前 2 周）
    列出所有你能引用的 Tier 1-3 报道。如果数量不够 5 条 + 时间跨度不够 2 年，先去做 PR：发媒体邮件 / Trade publication 投稿，养够 source pool 再写 wiki。
  3. 写 draft
    在 User:[你的账号]/sandbox 写，不直接发到主空间。完成后用 {{AfC submission}} 提交 Articles for Creation review。
  4. AfC 审稿（1-4 周）
    等 reviewer 通过。被拒就按反馈改——通常是引用质量。不要跳过 AfC 直接 move 到主空间——会被秒删。
  5. 上线后维护
    • 每季度跑一次 archive.org/save/ 让所有 cite 保活
    • 每次融资 / 重大产品发布配 2 条独立 Tier-1 报道后再补条目
    • Talk 页定期回应其他编辑者的修改建议
    • 同步更新 Wikidata Q-ID
  6. 建反向链接（防 orphan）
    找 5-10 个真实相关的现有 Wikipedia 文章加 wikilink 进来。每次加之前先确认该处事实有可靠来源支撑，否则被回滚。

怎么验证你的 wiki 被 LLM 抓到了

验证方法预期
Wikipedia 收录访问条目 URL 返回 200，无 {{Speedy deletion}}条目存活 > 30 天
Wikidata 关联wikidata.org/wiki/QXXXXXX 与条目双向链接有 P31 (instance of), P452 (industry)
Common Crawl 抓到commoncrawl.org search interface 查 URL下一个月度 dump 收录
ChatGPT 回忆用 GPT-4 / GPT-5 问 "What is [公司名]"能给出和 lead 一致的定义
Perplexity 引用perplexity.ai 同样问题Sources 列表里有你的 Wikipedia URL
Claude 引用claude.ai 同样问题能复述 infobox 字段

从 wiki 发布到 LLM 训练 cycle 吃进去：约 6-12 个月（Claude / GPT 主要在年度大版本更新时刷新 Wikipedia snapshot）。短期内可通过 Perplexity / GPT-4 web search 模式立即生效。
